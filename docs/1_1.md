# 발표 내용

## 서버는 무엇이고 어떻게 동작할까요?, 자바 웹 프레임워크는 어떻게 변화해 왔을까요?

* 서버와 클라이언트는 무엇인가요?
> 서비스를 제공하는 방향에 따라 서비스를 제공하는 서버와 서비스를 필요로 하고 사용하는 클라이언트로 나뉜다. 일반적으로 클라이언트는 필요한 서비스를 담당하는 서버에 요청하고, 서버는 요청에 대한 응답의 형태로 서비스를 제공한다. 네트워크 방식 중 하나인 P2P의 경우 각각의 Peer가 서비스의 요청과 제공을 모두 수행하기 때문에 서버-클라이언트와 구별되곤 한다.
* 자바 웹 프레임워크의 역사
  * J2EE(JavaEE), EJB, Servlet은 무엇일까요?
> J2EE는 Java 2 Enterprise Edition의 약자로 기업용 어플리케이션 개발을 위해 만들어진 자바 플랫폼이다. Java의 표준 플랫폼인 Java SE의 위에서 EJB나 Servlet처럼 여러 구성요소를 갖추어 웹, 기업 어플리케이션을 실행할 수 있는 환경을 제공한다. 본래 Java를 개발한 Sun에서 제작되고 제공되었지만 Eclipse로 개발이 넘어간 현재에는 Jakarta EE라는 이름으로 바뀌어 운영되고 있다.
> EJB는 기업이 서비스를 관리할 수 있는 J2EE의 구성 요소이다. 대규모 서비스를 관리하기 위한 분산 트랜잭션을 비롯한 여러 기능을 제공하는 서버를 위한 컴포넌트이다. 여러가지 Beans을 이용해 서비스의 생산과 관리를 돕지만 상대적으로 운영 성능이나 개발 생산성이 낮다.
> EJB가 서버에서 통신, 생명 주기 등의 다양한 관리를 담당한다고 하면 Servlet은 클라이언트의 http요청을 처리하기 위해 존재한다. 동적 페이지를 제공하기 위한 Java의 클래스로 request 객체와 response 객체를 통해 요청과 응답을 주고받는다.

  * 스프링 프레임워크는 어떻게 탄생하게 되었나요?
> J2EE 이후, Java를 사용한 기업 애플리케이션은 JSB를 사용하여 제작되었지만 JSB 자체의 문제점으로 어려움이 많았다. EJB로 로우 레벨 기술을 직접 관리할 필요는 줄었지만 컴포넌트를 위한 요구사항에 맞추기 위한 작업이 필요하게 되었다. 이로 인해 이로 인해 생산성도 줄고 제작된 코드 등이 EJB에 맞추어야 한다는 종속성도 강해졌다. 문제를 해결하기 위해선 더 가볍고 쉬운 프레임워크크가 필요하게 되었기 때문에 Spring이 등장하게 되었다.

## Spring Framework의 주요 특징은 무엇인가요? 그리고 그 특징이 어떻게 구현될 수 있었을까요?

* Spring Framework의 주요 특징은 무엇일까요?
  * DI는 무엇인가요? 
> DI는 의존성 주입으로 IoC로 인해 Spring이 관리한 관계에 따라 객체 간의 관계를 설정하는 것이다. 개발자에 코드를 확장하고 수정하더라도 변경된 관계에 따라 다른 관계가 새롭게 주입된다.
  * IoC는 무엇인가요?
>  IoC는 제어 역전이라고도 불리며 필요한 객체를 생성하고 Java 특유의 관계를 설정하는 데 도움을 준다. 자주 사용되는 상속에서 개발자는 다양한 상속관계를 계속 고려하면서 개발해야 했지만 Spring Framework가 관계를 대신 관리해주는 것이다. 단순히 관계뿐 아니라 객체 생성과 삭제와 같은 생명 주기도 관리해준다.
  * AOP는 무엇인가요?
> AOP는 관심 지향 프로그래밍으로 공통된 관심사에 대한 개발을 용이하게 하는 것이다. 서비스가 제공하는 기능이 다양할 수 있지만 각 기능은 유사하거나 동일한 작업을 필요로 할 수 있다. 다른 기능에 포함된 같은 작업을 하나의 관심사로 분류하고 개발하는 것이 AOP이다.

## Spring은 무엇일까요? 또, Spring과 SpringBoot의 주요한 차이점은 뭘까요?

* Spring은 무엇일까요?
> EJB를 대체하게 된 Spring은 마찬가지로 기업용 어플리케이션을 Java로 개발할 수 있게 해주는 프레임워크이다. 관련 규약을 강하게 지켜야 했던 EJB와 다르게 기본적인 Java만을 알고 있더라도 최대한 개발할 수 있도록 구성되어 있어 어플리케이션 개발 난이도가 더 쉽다. 오픈소스임에도 스프링소스라는 기업에서 개발과 관리가 이루어진다.
* Spring은 어떤 구성요소를 포함하고 있나요? 
> Spring은 기업 어플리케이션을 위한 다양한 요소를 포함하고 있다. ServletContainer는 http에 대한 처리를 담당하는 Servlet을 관리한다. Controller는 http request를 처리하는 요소이며 Service는 비즈니스 로직을, DAO는 Database에의 접근을 담당하는 요소이다. 앞의 3가지 요소는 개발자가 Java로 직접 개발하게 되는 컴포턴트이다. UI화면을 제공하는 View 또한 따로 존재한다.
* MVC 패턴은 어떤 패턴을 말할까요? 또, 이 패턴은 어떻게 동작하나요?
> 프로그램에서 Model과 View, Controller의 3 요소로 구성된 구조를 MVC 패턴이라고 한다. Model은 데이터와 비즈니스 로직의 처리를 담당하며 View는 사용자의 인터페이스를 담당한다. Controller는 Model과 View 사이에서 View가 받은 입력을 Model에, Model이 가져온 결과를 View에 전달하는 통신을 담당한다.
* Spring과 SpringBoot의 주요한 차이점은 무엇일까요?
> Spring으로 인해 서버 어플리케이션 개발은 쉬워졌지만 여전히 기본 설정이 어렵고 WAS 서버를 추가로 구성해야 했다. 더 빠르고 쉬운 개발을 위해 Spring의 프레임워크로 개발된 것이 SpringBoot이다. 기존에 필요했던 설정을 Spring은 자동으로 처리해주며 내장된 Tomcat과 같은 서버가 존재하기에 WAS를 별도로 추가하고 설정할 필요가 없다.

## Servlet Container와 DispatcherServlet, Application Context(Spring Container)는 무엇인가요? 그리고 어떻게 동작하나요?

* Servlet Container는 무엇인가요?
  * Servlet Container는 사용자의 요청을 어떻게 처리하나요?
> Servlet Container는 사용자의 요청에 대응해 Servlet의 실행과 처리를 관리한다. 사용자의 요청은 쓰래드를 만들며 Servlet이 연결될 request, response 객체가 Servlet Container에 의해 생성된다. 응답이 response 객체를 통해 다시 전달되면 작업이 끝난 객체를 제거하는 것이다.
* DispatcherServlet은 뭘까요?
  * 프론트 컨트롤러 패턴은 무엇일까요?
> 클라이언트마다 서버의 다른 컨트롤러가 직접 대응하는 것이 기존의 방식이었다. 프론트 컨트롤러는 클라이언트의 요청을 모두 모아 받고 특정한 상황에서 필요한 컨트롤러와 연결해준다. 통신을 위한 서블릿이 프론트 컨트롤러 하나로 통일되기 때문에 다른 컨트롤러는 서블릿을 부를 필요가 없어지고 공통된 부분은 다시 작성하지 않아도 된다. Spring에서는 DispatcherServlet이 프론트 컨트롤러를 담당한다.
* Application Context는 무엇을 의미할까요?
> Application Context는 Spring의 Bean을 생성, 관리한다. Spring의 특징 중 하나인 IoC 또한 Bean 객체와 관련되었기 때문에 이 특징을 지키고 관리하는 담당이 Application Context이다.
* SpringBoot는 어떻게 기동되나요?
  * SpringApplication.run()을 하면 어떤 일이 일어나나요?
    * SpringBoot, Servlet Container, Application Context(Spring Container), Dispatcher Servlet은 서로 어떻게 상호작용 하나요?
    > SpringBoot를 실행하면 기본적으로 필요한 요소들을 SpringBoot가 자동으로 설정한다. Bean 관리에 필요한 Application Context 또한 SpringBoot에 의해 생성되고 초기화된다.
    * Servlet Container에는 무엇이 탑재될까요?
